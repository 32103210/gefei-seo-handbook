# 第14章：域名注册与网站部署上线

本章将手把手教你在 30 分钟内完成从域名选购到网站正式上线的全部流程。建站的技术门槛已经大幅降低，但你需要对 DNS、CDN、SSL/HTTPS 等核心概念有清晰理解，才能在出问题时快速定位原因。

---

## 14.1 域名选择：优先级与策略

域名是网站的门牌号，直接影响用户信任度和搜索引擎评分。选域名的核心原则只有一条：**简短、好记、与内容相关**。

### 14.1.1 后缀优先级

不同顶级域名（TLD）在用户认知和 SEO 效果上有明显差异。建议按以下优先级选择：

| 优先级 | TLD | 说明 | 适用场景 |
|--------|-----|------|----------|
| 第1优先 | `.com` | 全球认知度最高，用户自然记忆 | 商业网站、主站 |
| 第2优先 | `-com`（双拼） | 中文语境下便于输入 | 中文内容站 |
| 第3优先 | `.net` | 历史上用于网络技术类网站 | 技术博客、社区 |
| 第4优先 | `.org` | 非营利组织常用 | NGO、教育类站点 |
| 第5优先 | `.io` | 科技圈流行 | 开发者工具、SaaS |
| 其他 | `.xyz`、`.info` 等 | 搜索引擎信任度较低 | 谨慎使用 |

`.com` 永远是第一选择。如果你的目标词对应的 `.com` 域名已被注册，按以下顺序考虑：

1. **联系域名持有者购买**（适合关键词域名，价值较高时值得投入）
2. **选择同义词的 `.com`**（比如 "seo" 被注册就试试 "seoer.com"）
3. **考虑 `-com` 双拼域名**（如 "zhushou.com"，国内用户输入习惯友好）
4. **最后才选 `.net` 或其他后缀**

### 14.1.2 域名命名原则

```
✅ 好的域名示例：
  - meijin.com（金的，含义明确）
  - yibaiyang.com（一百样，电商品牌）
  - tuike.io（推客，科技感）

❌ 差的域名示例：
  - my-best-seo-tool-guide-2024.com（过长，难以记忆和输入）
  - xz123456.net（无意义字符）
  - chinese-food-near-me.net（中间包含连词符）
```

**实操建议**：在 GoDaddy、Namecheap 或国内的阿里云（万网）、腾讯云 DNSPod 上搜索目标域名。价格通常在 60～150 元/年，`.io` 域名可能更贵。

### 14.1.3 域名购买后的第一件事

购买域名后，立即记录以下信息：
- 域名注册商（如阿里云、GoDaddy）
- 登录账号和密码（建议使用密码管理器）
- 域名到期时间（提前30天续费）
- 注册时填写的 WHOIS 信息（建议开启隐私保护）

## 14.2 域名解析服务商：为什么选 Cloudflare

Cloudflare 是全球最大的 CDN 和 DNS 服务商之一，提供免费套餐，对个人站长和小型项目来说功能十分完整。核心优势：

- **全球分布式 DNS**：访问速度更快、更稳定
- **免费 HTTPS/CDN**：无需额外付费即可获得全站加速
- **DDoS 防护**：基础防护免费开启
- **功能丰富**：HTTPS 重写、自动跳转、页面规则等

## 14.3 Cloudflare 完整配置流程

### 14.3.1 注册与添加站点

1. 访问 [https://dash.cloudflare.com](https://dash.cloudflare.com)，使用邮箱注册账号
2. 点击 "Add a Site"，输入你的域名（如 `example.com`），点击 "Add Site"
3. 选择 **Free** 计划（免费版已包含绝大多数你需要的功能），点击 "Continue"
4. Cloudflare 会扫描你域名现有的 DNS 记录，核对后点击 "Continue"

### 14.3.2 修改域名注册商的 Nameservers

这是最关键的一步，必须在域名注册商处将默认 DNS 服务器替换为 Cloudflare 分配的地址。

Cloudflare 通常会给你两个 nameserver 地址，格式类似：
```
ns3.cloudflare.com
ns4.cloudflare.com
```

**在阿里云（万网）修改方法：**
1. 登录阿里云控制台 → 域名与网站（万网）→ 我的域名
2. 点击域名右侧的 "管理" → "DNS修改"
3. 填写 Cloudflare 给出的两个 nameserver 地址，提交

**在 GoDaddy 修改方法：**
1. 登录 GoDaddy → My Products → DNS → Change
2. 选择 "I'll use my own nameservers"
3. 输入 Cloudflare 的两个地址，保存

> **重要**：修改 nameserver 后，生效时间可能需要 5 分钟到 48 小时。期间网站可能出现短暂不可访问，属正常现象。

### 14.3.3 开启关键安全与性能功能

Nameserver 更替完成后，重新登录 Cloudflare 控制台，进入你站点的管理面板，按以下方式配置：

#### 自动 HTTPS 重写（Auto HTTPS Rewrites）

**位置**：SSL/TLS → Edge Certificates → Automatic HTTPS Rewrites

**作用**：自动将页面中所有 HTTP 链接转换为 HTTPS，避免"混合内容"（Mixed Content）警告。开启后，浏览器地址栏将稳定显示绿色锁标。

#### 始终使用 HTTPS（Always Use HTTPS）

**位置**：SSL/TLS → Edge Certificates → Always Use HTTPS

**作用**：将所有 HTTP 请求 301 重定向到 HTTPS。**只要你的 SSL 证书配置正确，此选项务必开启**。

#### Brotli 压缩

**位置**：Speed → Optimization → Content Optimization → Brotli

**作用**：比 Gzip 压缩率更高的文本压缩格式，可减少 15%～25% 的传输体积，显著提升页面加载速度。

### 14.3.4 SSL/TLS 模式选择（最重要）

**位置**：SSL/TLS → Overview

Cloudflare 提供四种 SSL/TLS 模式，正确选择直接影响网站安全性：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **Off** | 无加密 | 绝对不要用 |
| **Flexible** | 浏览器到 Cloudflare 加密，Cloudflare 到源站不加密 | 仅作为临时方案 |
| **Full** | 全链路加密，Cloudflare 验证 SSL 证书但不自签名 | **推荐选择** |
| **Full (strict)** | 全链路加密，Cloudflare 到源站也必须是有效证书 | 最佳安全方案 |

**必须选择 "Full" 或 "Full (strict)"，绝对不能选 "Flexible"。**

"Flexible" 模式意味着 Cloudflare 与你源服务器之间的通信是明文的，如果你的源服务器 IP 被攻击者获取，流量内容可被窃听。这是新手最常犯的错误配置之一。

## 14.4 使用 Git 管理代码

### 14.4.1 创建 GitHub 仓库

1. 登录 [https://github.com](https://github.com)，点击右上角 "+" → "New repository"
2. 填写仓库信息：
   - **Repository name**：`my-seo-site`（用实际项目名替换）
   - **Description**：简要描述项目
   - **Public/Private**：选择 Public（公开）或 Private（私有）
   - 勾选 "Add a README file"（推荐）
3. 点击 "Create repository"

### 14.4.2 本地初始化与首次提交

在你的网站项目文件夹中打开终端，执行以下命令：

```bash
# 进入项目目录
cd /path/to/your/website

# 初始化 Git 仓库
git init

# 配置用户名和邮箱（仅首次使用需要）
git config user.name "你的GitHub用户名"
git config user.email "你的GitHub注册邮箱"

# 添加所有文件到暂存区
git add .

# 提交到本地仓库
git commit -m "Initial commit: 添加网站初始文件"

# 添加远程仓库地址（替换为你的实际仓库URL）
git remote add origin https://github.com/你的用户名/仓库名.git

# 推送到 GitHub
git branch -M main
git push -u origin main
```

> 如果推送时提示认证失败，需要先在 GitHub 生成 Personal Access Token 或配置 SSH Key。推荐使用 HTTPS + Token 方式，操作更简单。

### 14.4.3 日常代码更新流程

```bash
# 查看当前状态
git status

# 添加修改的文件
git add .

# 提交（养成习惯，每次提交说明清楚改了什么）
git commit -m "修复首页标题标签缺失问题"

# 推送到远程
git push
```

## 14.5 部署到 Vercel

Vercel 是目前最适合部署静态网站和 Next.js 应用的平台，与 GitHub 深度集成，提交代码后自动触发构建和部署，体验极为流畅。

### 14.5.1 注册 Vercel

1. 访问 [https://vercel.com](https://vercel.com)
2. 点击 "Sign Up"，选择用 GitHub 账号授权登录（推荐）
3. 完成基础信息填写，进入控制台

### 14.5.2 从 GitHub 导入项目

1. 在 Vercel 控制台，点击 "Add New..." → "Project"
2. 在 "Import Git Repository" 页面，找到你的 GitHub 仓库，点击 "Import"
3. Vercel 会自动检测项目框架（如果是静态 HTML，Framework Preset 选择 "Other"）
4. 配置构建命令和输出目录：
   - **Build Command**：（静态网站留空）
   - **Output Directory**：`/`（根目录）
5. 点击 "Deploy"，等待首次部署完成（通常 30 秒内）

### 14.5.3 自定义域名配置（关键步骤）

部署成功后，Vercel 会给你一个默认域名（如 `your-project.vercel.app`）。配置自定义域名的步骤如下：

**在 Vercel 端：**
1. 进入你的项目 → Settings → Domains
2. 在输入框中填写你的域名（如 `example.com`），点击 "Add"
3. Vercel 会提示需要在 DNS 中添加哪些记录，**先不要关闭此页面**

**在 Cloudflare 端添加 DNS 记录：**

进入 Cloudflare 控制台 → 你的站点 → DNS → Records，点击 "Add record"：

```
# www 子域名：使用 CNAME 记录
类型：CNAME
名称：www
目标：cname.vercel-dns.com（Vercel 提供的 CNAME 地址）
代理状态：DNS only（关闭代理）或 Proxied（开启CDN）

# 根域名：使用 A 记录（Cloudflare 不支持根域名 CNAME，需用 A 记录）
类型：A
名称：@
目标：76.76.21.21（Vercel 提供的 IP 地址）
代理状态：DNS only 或 Proxied
```

> **为什么根域名用 A 记录而不是 CNAME？** DNS 规范不允许在根域名（bare domain，如 `example.com`）上设置 CNAME 记录。Cloudflare 虽然通过特殊处理允许这样做，但最标准的做法仍是使用 A 记录指向 Vercel 的固定 IP `76.76.21.21`。

### 14.5.4 等待 SSL 证书自动签发

Vercel 会自动为你的自定义域名申请和续期 SSL 证书。回到 Vercel Domains 页面，等待域名状态从 "Pending" 变为 "Valid"（通常 1～5 分钟）。

如果状态一直是 "Renewal Pending"，点击 "Check Again" 刷新。如果超过 15 分钟仍未生效，检查：
1. Cloudflare DNS 记录是否正确添加
2. Cloudflare 的 nameserver 是否已完全生效

## 14.6 完整 DNS 配置汇总

部署完成后，你的 Cloudflare DNS 记录应该如下所示（以 `example.com` 为例）：

| 类型 | 名称 | 内容 | 代理状态 |
|------|------|------|----------|
| A | @ | 76.76.21.21 | Proxied |
| CNAME | www | cname.vercel-dns.com | Proxied |
| TXT | @ | （用于验证域名所有权，如 Google Search Console） | DNS only |

## 14.7 常见问题与解决方案

### 问题一：部署后网站显示 "SSL handshake failed"

**原因**：SSL/TLS 模式设置为 "Flexible" 或未正确配置。

**解决**：
1. 进入 Cloudflare → SSL/TLS → Overview
2. 将模式切换为 **"Full"**
3. 等待 1～2 分钟后刷新页面

### 问题二：Vercel 提示域名已被其他账户使用

**原因**：该域名之前在其他 Vercel 账户配置过。

**解决**：
1. 如果你拥有该域名：在域名注册商处添加一条 TXT 记录，Vercel 会通过该记录验证你对该域名的所有权
2. 联系 Vercel Support，说明情况，提供域名所有权证明
3. 如果是误配置：检查是否无意中在其他 Vercel 账户添加过该域名

### 问题三：访问域名显示 "522 - Connection timed out"

**原因**：源站（Vercel）IP 被 Cloudflare 防火墙拦截，或 Vercel 未正确接收域名流量。

**解决**：
1. 检查 Cloudflare SSL/TLS 模式是否为 Full
2. 在 Cloudflare → Security → WAF 中检查是否有异常规则触达
3. 在 Vercel Domains 页面确认域名状态为 Valid

### 问题四：开启了 Cloudflare CDN 但网站变慢

**原因**：代理状态设置为 "Proxied" 但源站位于中国大陆。

**解决**：
- 如果目标用户在中国大陆，将源站域名的代理状态改为 **DNS only**（灰色云朵图标）
- Cloudflare 在中国大陆没有足够的边缘节点，国内访问反而可能更慢

### 问题五：GitHub 推送被拒绝

**原因**：本地仓库与远程仓库历史不一致。

**解决**：
```bash
# 方法1：拉取远程内容再推送
git pull origin main --rebase
git push origin main

# 方法2：强制推送（小心使用，会覆盖远程历史）
git push -u origin main --force
```

## 14.8 本章小结

域名注册与部署上线的完整流程总结如下：

```
1. 选购域名（优先 .com）→ 2. 注册 Cloudflare 并添加站点 → 3. 修改域名注册商 Nameserver → 4. 配置 Cloudflare 安全设置（Full SSL、Brotli）→ 5. 在 GitHub 创建仓库并推送代码 → 6. 在 Vercel 导入 GitHub 仓库并部署 → 7. 在 Vercel 添加自定义域名 → 8. 在 Cloudflare 添加 DNS 记录 → 9. 等待 SSL 证书生效 → 10. 测试访问
```

整条链路中最容易出问题的是两个环节：**Nameserver 修改**（影响整个 DNS 解析）和 **SSL 模式选择**（影响 HTTPS 访问）。记住 "Full" 而非 "Flexible"，问题就解决了一大半。
